## 1. if는 표현식이다

> **선생님**: Rust의 `if`는 값을 만들어내는 표현식이에요.
>
> ```rust
> // 일반적인 if
> if score > 90 {
>     println!("A");
> } else if score > 80 {
>     println!("B");
> } else {
>     println!("C");
> }
>
> // Rust만의 방식 — if를 값으로 사용
> let grade = if score > 90 { "A" } else { "B" };
> ```
>
> 주의: **양쪽 타입이 같아야 해요.**
>
> ```rust
> let x = if true { 5 } else { "hello" };  // 에러! i32와 &str가 다름
> ```
>
> 컴파일러가 `x`의 타입을 하나로 확정해야 하니까요.

**🙋 학생**: 컴파일 타임에 잡자라는 부분은 왜 계속 강조되는 거죠?

> **선생님**: 실무적으로 말하면:
> - **런타임에 터지면** → 서비스가 죽어요. 유저가 피해를 봐요.
> - **컴파일 타임에 잡으면** → 빌드가 안 돼요. 개발자가 바로 고쳐요.
>
> 지금 하시는 업무에 연결하면 — C++은 메모리 실수가 런타임에 터져요 (segfault, double free, OOM). 그래서 메모리 샌드박스로 격리하려는 거잖아요. Rust는 그런 실수를 대부분 컴파일할 때 잡아줘요.
>
> 불변 기본, 타입 명시, 표현식 중심 — 다 "런타임에 터질 걸 컴파일 타임으로 끌어오자"의 구체적인 방법들이에요.

### 💡 핵심
- `if`는 표현식 → 값으로 사용 가능
- 양쪽 타입이 같아야 함
- 컴파일 타임에 잡자 = 런타임 사고 방지

## 2. 반복문 3가지 — loop, while, for

> **선생님**: Rust에 반복문이 3가지 있어요.
>
> **1. `loop` — 무한 반복 (직접 break로 탈출)**
>
> ```rust
> let mut count = 0;
> loop {
>     count += 1;
>     if count == 3 { break; }
> }
> ```
>
> `loop`도 표현식! `break`에 값을 붙이면 반환:
>
> ```rust
> let result = loop {
>     count += 1;
>     if count == 10 {
>         break count * 2;  // 20이 result에 들어감
>     }
> };
> ```
>
> **2. `while` — 조건이 참인 동안**
>
> ```rust
> let mut n = 3;
> while n > 0 {
>     println!("{n}");
>     n -= 1;
> }
> ```
>
> **3. `for` — 범위나 컬렉션 순회 (가장 많이 씀)**
>
> ```rust
> for i in 0..5 {        // 0, 1, 2, 3, 4
>     println!("{i}");
> }
>
> for i in 0..=5 {       // 0, 1, 2, 3, 4, 5 (=붙이면 끝 포함)
>     println!("{i}");
> }
> ```

### 💡 핵심
- `loop` = 무한 반복, break로 탈출, 값 반환 가능
- `while` = 조건부 반복
- `for` = 범위/컬렉션 순회 (실무에서 90%)
- `0..5` = 0~4, `0..=5` = 0~5

## 3. loop가 왜 존재하나 — while true와의 차이

**🙋 학생**: loop라는건 굳이 왜 생긴거야?

> **선생님**: 동작은 `while true`랑 같지만, **컴파일러 입장**에서 달라요.
>
> ```rust
> // while true — 컴파일러: "while은 안 돌 수도 있는 구조"
> let result;
> while true {
>     result = 42;
>     break;
> }
> println!("{result}");  // 에러 가능!
>
> // loop — 컴파일러: "무조건 진입하는 구조"
> let result = loop {
>     break 42;
> };
> println!("{result}");  // OK!
> ```

**🙋 학생**: 컴파일러가 확신을 못한다는게 무슨말이지?

> **선생님**: 컴파일러는 코드를 **문법 구조로만 판단**해요. 값을 실제로 실행해보지 않아요.
>
> `while 조건 { }` → 문법 구조 자체가 "안 돌 수 있는 것"으로 분류
> `loop { }` → 문법 구조 자체가 "무조건 도는 것"으로 분류
>
> `true`인지 변수인지 구분 안 하고, **문법 구조 자체**로 판단하는 거예요.

**🙋 학생**: 근데 컴파일러 입장에서 그게 왜 그렇게 중요해서 문법까지 만든거야?

> **선생님**: 컴파일러가 값을 분석하기 시작하면 복잡해져요. `true`는 쉽지만, `while some_flag`는? `while config.enabled`는? 어디까지 "확실히 true"라고 판단할 건지 경계가 모호해져요.
>
> Rust는 **문법 구조로 딱 나누는 방식**을 선택한 거예요. 단순하고 확실하게. 사소한 규칙이 쌓여서 "빌드 통과하면 안전하다"라는 신뢰가 만들어지는 거예요.

**🙋 학생**: 그럼 while문은 지양해야 하나? 아예 없애버리지 왜 나둔거지?

> **선생님**: 아니요, `while`은 제 역할이 있어요.
>
> ```rust
> // while이 맞는 상황 — 안 돌 수도 있는 반복
> while user_input != "quit" {
>     process(user_input);
> }
>
> // loop가 맞는 상황 — 무조건 돌다가 끊는 반복
> loop {
>     let data = fetch_request();
>     if shutdown_signal { break; }
>     handle(data);
> }
> ```
>
> 없앤 게 아니라, 각각 **의도가 다른 거**예요. "의도에 맞는 도구를 골라 쓰게" 하는 거예요.

**🙋 학생**: 그럼 while문을 선택하면 컴파일러가 값을 분석할 이유가 있다라고 판단하게 되니까 명시적으로 그렇게 하는게 맞으면 써라?

> **선생님**: 정확해요!
> - `while` 썼다 → "이건 안 돌 수도 있어요" 라고 컴파일러에게 말한 거
> - `loop` 썼다 → "이건 무조건 돌아요" 라고 말한 거
>
> 개발자가 문법을 선택하는 것 자체가 컴파일러에게 의도를 전달하는 방법인 거죠.

## Ch04 총정리

| 개념 | 의도 | 컴파일러 해석 |
|------|------|-------------|
| `if/else` | 조건 분기 (표현식) | 양쪽 타입 일치 검사 |
| `loop` | 무한 반복 | 반드시 진입한다고 판단 |
| `while` | 조건부 반복 | 안 돌 수도 있다고 판단 |
| `for` | 범위/컬렉션 순회 | 가장 안전, 가장 많이 사용 |

> **Rust 설계 원칙**: 문법 선택 = 컴파일러에게 의도 전달. 의도가 명확할수록 컴파일러가 더 많이 검사해줌.

## 번외: 왜 Java는 while(true)로 충분한데 Rust는 loop를 만들었나

**🙋 학생**: 자바는 왜 while문으로 무한루프를 만들어도 된다고 판단한거야?

> **선생님**: 자바는 **런타임에 잡는 철학**이거든요. JVM이 실행 중에 문제가 생기면 예외를 던져요. 런타임이 안전장치 역할을 해주니까, 컴파일러가 빡빡할 필요가 없었던 거예요.
>
> | | Java | Rust |
> |---|---|---|
> | 안전장치 | JVM (런타임) | 컴파일러 (빌드 타임) |
> | null 접근 | 런타임에 NPE | 컴파일 타임에 차단 (null 자체가 없음) |
> | 메모리 | GC가 알아서 정리 | 소유권으로 컴파일 타임에 관리 |
>
> **자바**: "일단 돌려, 문제 생기면 런타임이 잡아줄게"
> **Rust**: "빌드 단계에서 다 잡아, 런타임에는 문제 없게"

## 번외: "컴파일 타임에 잡자"가 왜 그렇게 중요한가 — C++/Java/Rust 비교

**🙋 학생**: 그래서 진짜 그런게 필요하다고? 누군가한테 설득할 수 있을 정도로 와닿진 않아.

> **선생님**: 실제 시나리오로 보여드릴게요.
>
> ### C++ — 아무것도 안 해줌
>
> ```cpp
> std::string name = "철수";
> std::string& read = name;    // 읽기 참조
> name += "님";                 // 수정 — 아무도 안 막음
> // 빌드 OK, 대부분 OK, 가끔 💥
> ```
>
> - GC도 없고, 컴파일러 검사도 없음
> - 개발자가 직접 mutex로 막아야 함
> - 까먹으면 런타임에 터짐, 재현 어려움
>
> ### Java — GC가 런타임에 알아서 관리
>
> ```java
> String name = "철수";
> String read = name;           // 참조 복사
> name = "철수님";              // 새 객체 생성, 이전 건 GC가 정리
> System.out.println(read);     // "철수" — 안전
> ```
>
> - GC가 런타임에 추적해서 정리
> - 메모리 문제 거의 안 터짐
> - 대신 **GC가 돌 때 프로그램이 잠깐 멈춤** (Stop-the-World)
>
> ### Rust — 컴파일러가 빌드 때 잡음
>
> ```rust
> let mut name = String::from("철수");
> let read = &name;            // 읽기 참조
> name.push_str("님");         // 컴파일 에러!
> println!("{read}");
> ```
>
> - GC 없음 → 멈춤 없음 → C++ 급 성능
> - 컴파일러가 참조 추적 → Java급 안전성
>
> | | 안전성 | 성능 | 대가 |
> |---|---|---|---|
> | C++ | ❌ 개발자 책임 | ✅ 최고 | 런타임 버그 디버깅 |
> | Java | ✅ GC가 관리 | ⚠️ GC 멈춤 | 성능 손해 |
> | Rust | ✅ 컴파일러가 관리 | ✅ 최고 | 빌드가 빡빡함 |
>
> **Rust가 "C++의 성능 + Java의 안전성"을 둘 다 가져가겠다는 거예요.**
> wasmtime을 Rust로 만드는 이유가 여기에 있어요 — GC 멈춤 없이 메모리 안전이 필요하니까.

**🙋 학생**: 자바에서도 그냥 참조 검사를 하면 되는 문제 아닌가? 왜 안 하는거지?

> **선생님**: 자바는 GC가 있어서 메모리 문제가 대부분 안 터져요. 이미 런타임에서 해결되니까 컴파일러가 빡빡할 필요가 없었던 거예요.
> Rust는 GC가 없으니까 컴파일러가 대신 빡빡해진 거고요.

## 번외: Rust 빌림 규칙 맛보기 (ch05 예고)

**🙋 학생**: Rust에서 참조 걸린 변수를 수정하고 싶으면 어떻게 해?

> **선생님**: 참조가 끝나면 돼요.
>
> ```rust
> let mut name = String::from("철수");
>
> let read = &name;
> println!("{read}");       // 여기서 read 사용 끝
>
> name.push_str("님");      // OK! read가 더 이상 안 쓰이니까
> ```
>
> 컴파일러는 **참조가 마지막으로 쓰인 지점**까지만 "읽는 중"으로 판단해요.

**🙋 학생**: 그 사이에 여러번 쓰는건 상관없고?

> **선생님**: 네, 읽기만 하면 몇 번이든 OK.
>
> ```rust
> let mut name = String::from("철수");
>
> let a = &name;
> let b = &name;
> let c = &name;
> println!("{a} {b} {c}");   // 다 읽기니까 OK
>
> name.push_str("님");       // OK! a, b, c 다 끝났으니까
> ```
>
> 규칙은 하나: **읽는 놈이 하나라도 살아있으면 수정 금지. 다 끝나면 수정 가능.**

### 💡 빌림 규칙 요약 (ch05에서 자세히)
1. **읽기 참조(`&`)는 여러 개 OK** — 다들 읽기만 하면 안전
2. **수정 참조(`&mut`)는 딱 하나만** — 그동안 읽기 참조도 없어야 함
3. 참조의 수명 = **마지막 사용 지점**까지
