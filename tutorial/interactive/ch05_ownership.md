# Ch05: 소유권 — 대화형 튜토리얼

---

## 1. 소유권이란?

> **선생님**: Rust에서 모든 값에는 **주인(owner)**이 딱 하나 있어요.
>
> ```rust
> let s = String::from("hello");  // s가 "hello"의 주인
> ```
>
> 규칙은 세 줄이에요:
> 1. 모든 값에는 **소유자(owner)**가 있다
> 2. 소유자는 **동시에 하나**만 존재한다
> 3. 소유자가 스코프를 벗어나면 **값이 자동으로 버려진다(drop)**

**🙋 학생**: 그게 C++의 소멸자(destructor)랑 비슷한 거야?

> **선생님**: 정확해요! C++에서 RAII 패턴으로 소멸자에서 자원 해제하는 거랑 같은 개념이에요. 차이점은 Rust는 **컴파일러가 소유권을 추적**해서, 누가 해제할지 헷갈릴 일이 없다는 거예요.
>
> ```rust
> {
>     let s = String::from("hello");  // s가 주인
>     println!("{s}");                 // s 사용
> }   // ← 여기서 s가 스코프를 벗어남 → 자동으로 drop (메모리 해제)
> ```
>
> C++에서는 delete 잊어버리면 메모리 누수, 두 번 하면 double free. Rust에서는 컴파일러가 알아서 딱 한 번 정리해줘요.

### 💡 핵심
- 모든 값에는 소유자가 딱 하나
- 소유자가 스코프를 벗어나면 자동 drop
- C++의 RAII + 소멸자와 같은 개념이지만, 컴파일러가 강제

## 2. 이동(Move) — 주인이 바뀐다

> **선생님**: 변수를 다른 변수에 대입하면 **소유권이 이동**해요.
>
> ```rust
> let s1 = String::from("hello");
> let s2 = s1;   // 소유권이 s1 → s2로 이동!
> // println!("{s1}");  // 컴파일 에러! s1은 이제 빈 껍데기
> println!("{s2}");     // OK — s2가 새 주인
> ```

**🙋 학생**: 왜 복사가 아니라 이동이야? C++에서는 그냥 복사되잖아.

> **선생님**: `String`은 힙 메모리를 쓰는 타입이에요. 만약 복사하면 힙 데이터를 통째로 복제해야 하는데, 그건 비쌀 수 있어요.
>
> ```
> 스택         힙
> s1: [ptr, len, cap] → "hello" (힙 메모리)
>
> let s2 = s1; 이후:
> s1: (무효화)
> s2: [ptr, len, cap] → "hello" (같은 힙 메모리)
> ```
>
> 두 변수가 같은 힙 메모리를 가리키면 위험하잖아요? 둘 다 스코프 끝에서 해제하면 **double free**. 그래서 Rust는 "이동시키고 원래 주인은 무효화"하는 전략을 택한 거예요.

**🙋 학생**: 그러면 진짜 복사하고 싶으면 어떻게 해?

> **선생님**: `clone()`을 명시적으로 호출하면 돼요.
>
> ```rust
> let s1 = String::from("hello");
> let s2 = s1.clone();   // 힙 데이터까지 전부 복사
> println!("{s1}");       // OK! s1도 여전히 유효
> println!("{s2}");       // OK! s2는 독립적인 복사본
> ```
>
> **"비싼 복사는 명시적으로"** — Rust의 철학이에요. 실수로 비용이 큰 복사가 일어나는 걸 막아줘요.

### 💡 핵심
- 대입(`=`)하면 소유권이 **이동(move)** — 원래 변수는 무효
- 이유: 같은 힙 메모리를 두 변수가 가리키면 double free 위험
- 진짜 복사가 필요하면 `clone()` 명시 호출
- "비싼 복사는 명시적으로" = Rust 철학

## 3. Copy 트레이트 — 스택에 사는 타입들

> **선생님**: 근데 아까 ch02에서 이런 코드 본 적 있죠?
>
> ```rust
> let x = 5;
> let y = x;    // x가 이동되어야 할 텐데...
> println!("{x}");  // 어? 잘 되네?
> ```
>
> 정수, 불리언, 부동소수점, char 같은 **스택에만 사는 타입**은 복사해도 거의 비용이 없어요. 그래서 Rust가 자동으로 복사해줘요.

**🙋 학생**: 어떻게 구분하지? 어떤 타입이 이동이고 어떤 게 복사야?

> **선생님**: `Copy` 트레이트를 가진 타입은 자동 복사, 아니면 이동이에요.
>
> | Copy (자동 복사) | Move (이동) |
> |-----------------|------------|
> | `i32`, `u64`, `f64` 등 정수/실수 | `String` |
> | `bool` | `Vec<T>` |
> | `char` | `Box<T>` |
> | 튜플 (원소가 전부 Copy일 때) | 구조체 (기본) |
>
> 규칙이 간단해요: **힙을 쓰면 Move, 스택만 쓰면 Copy.**

**🙋 학생**: C++에서는 어떻게 처리해?

> **선생님**: C++은 기본이 **전부 복사**예요. 이동하고 싶으면 `std::move()`를 명시적으로 써야 하죠. Rust는 반대예요 — **기본이 이동**, 복사하고 싶으면 `clone()`. 정반대 철학이에요.
>
> | | C++ | Rust |
> |--|-----|------|
> | 기본 동작 | 복사 (copy constructor) | 이동 (move) |
> | 이동하려면 | `std::move()` 명시 | 자동 |
> | 복사하려면 | 자동 | `clone()` 명시 |
> | 안전성 | 개발자 책임 | 컴파일러 보장 |

### 💡 핵심
- `Copy` 트레이트 = 스택 전용 타입, 자동 복사 (`i32`, `bool`, `char` 등)
- 나머지 = 이동(Move). 힙을 쓰는 타입은 대입 시 이동
- C++은 기본 복사, Rust는 기본 이동 — 정반대 철학
- 간단 판별: 힙 쓰면 Move, 스택만 쓰면 Copy

## 4. 함수와 소유권

> **선생님**: 함수에 값을 넘기면 **대입과 똑같이** 소유권이 이동해요.
>
> ```rust
> fn take_ownership(s: String) {
>     println!("{s}");
> }   // s가 여기서 drop
>
> let name = String::from("철수");
> take_ownership(name);
> // println!("{name}");  // 에러! name의 소유권은 함수 안으로 이동됨
> ```

**🙋 학생**: 함수 호출할 때마다 소유권이 가버리면 불편하지 않아?

> **선생님**: 맞아요, 그래서 **참조(reference)**를 써요. 소유권을 넘기지 않고 **빌려주는** 거예요.
>
> ```rust
> fn print_length(s: &String) {  // &String = String의 참조를 받음
>     println!("길이: {}", s.len());
> }   // s는 참조일 뿐이라 drop 안 됨
>
> let name = String::from("철수");
> print_length(&name);       // &name = name을 빌려줌
> println!("{name}");        // OK! name은 여전히 주인
> ```
>
> `&`가 "빌림"이에요. 빌려준 거니까 소유권은 원래 주인에게 그대로 있어요.

**🙋 학생**: 근데 빌린 거로 값을 수정하고 싶으면?

> **선생님**: `&mut`으로 **수정 가능한 빌림**을 하면 돼요.
>
> ```rust
> fn add_suffix(s: &mut String) {
>     s.push_str("님");
> }
>
> let mut name = String::from("철수");
> add_suffix(&mut name);
> println!("{name}");  // "철수님"
> ```
>
> 정리하면:
> - `&T` = 읽기 전용 빌림 (immutable reference)
> - `&mut T` = 수정 가능 빌림 (mutable reference)

### 💡 핵심
- 함수 인자 전달 = 대입과 동일 (소유권 이동)
- `&T`: 읽기 빌림 — 소유권 유지, 값 수정 불가
- `&mut T`: 수정 빌림 — 소유권 유지, 값 수정 가능
- 빌림 = 소유권을 넘기지 않고 잠시 쓰게 해주는 것

## 5. 빌림 규칙 — 컴파일러의 심판

> **선생님**: ch04에서 맛보기로 봤던 빌림 규칙, 이제 정식으로 배울게요. 규칙은 딱 두 개예요.
>
> **규칙 1: 읽기 참조(`&`)는 여러 개 동시 OK**
>
> ```rust
> let s = String::from("hello");
> let r1 = &s;
> let r2 = &s;
> let r3 = &s;
> println!("{r1} {r2} {r3}");  // OK! 다 읽기니까
> ```
>
> **규칙 2: 수정 참조(`&mut`)는 동시에 딱 하나. 그리고 그동안 읽기 참조도 없어야 함**
>
> ```rust
> let mut s = String::from("hello");
> let r = &mut s;
> r.push_str(" world");
> // let r2 = &s;  // 에러! &mut이 살아있는 동안 &도 안 됨
> println!("{r}");
> ```

**🙋 학생**: 왜 그렇게 빡빡해?

> **선생님**: 데이터 경합(data race)을 **원천 차단**하기 위해서예요.
>
> 데이터 경합이 일어나려면 3가지가 동시에 필요해요:
> 1. 두 개 이상의 포인터가 같은 데이터에 접근
> 2. 그중 하나 이상이 데이터를 **수정**
> 3. 접근 순서를 보장하는 메커니즘이 없음
>
> Rust의 빌림 규칙은 2번을 컴파일 타임에 차단해요. 수정하는 놈이 있으면 다른 누구도 동시에 접근 못 하게.

**🙋 학생**: "살아있다"의 기준이 뭐야? 변수가 선언된 시점부터 스코프 끝까지?

> **선생님**: 아뇨, Rust 컴파일러는 **NLL (Non-Lexical Lifetimes)**이라는 걸 써요. 참조가 **마지막으로 사용된 지점**까지만 "살아있다"고 판단해요.
>
> ```rust
> let mut s = String::from("hello");
>
> let r = &s;
> println!("{r}");        // r의 마지막 사용 ← 여기서 r의 수명 끝
>
> s.push_str(" world");   // OK! r은 이미 끝남
> println!("{s}");
> ```
>
> 스코프(`{}`)가 아니라 **실제 사용 범위**를 추적하는 거예요. 그래서 생각보다 유연해요.

### 💡 핵심
- 규칙 1: `&T` 여러 개 동시 OK (읽기끼리는 안전)
- 규칙 2: `&mut T`는 독점. 살아있는 동안 다른 참조 불가
- 이유: 데이터 경합(data race) 원천 차단
- NLL: 참조 수명 = 마지막 사용 지점까지 (스코프 끝이 아님)

## 6. 소유권 패턴 정리 — 언제 뭘 쓸까

> **선생님**: 실무에서 헷갈리는 건 "그래서 언제 뭘 써야 해?"죠. 정리해드릴게요.
>
> | 상황 | 사용 | 예시 |
> |------|------|------|
> | 값을 읽기만 할 때 | `&T` | `fn print(s: &String)` |
> | 값을 수정해야 할 때 | `&mut T` | `fn append(s: &mut String)` |
> | 값을 소비(consume)할 때 | `T` (소유권 이동) | `fn drop_it(s: String)` |
> | 값을 복사하고 싶을 때 | `.clone()` | `let s2 = s1.clone()` |
>
> **90%의 경우 `&T`(읽기 빌림)를 쓰면 됩니다.** 함수가 값을 소비할 필요가 없다면 참조를 받는 게 맞아요.

**🙋 학생**: 소유권 에러가 나면 어떻게 대처해?

> **선생님**: 단계적으로 접근하면 돼요:
>
> 1. **"이 값을 여기서 꼭 소유해야 하나?"** → 아니면 `&`로 빌리기
> 2. **"수정이 꼭 필요한가?"** → 아니면 `&T`, 필요하면 `&mut T`
> 3. **"두 곳에서 동시에 필요한가?"** → `clone()`으로 복사 (성능 비용 있음)
> 4. **"정말 공유 소유권이 필요한가?"** → `Rc<T>`, `Arc<T>` (나중에 배움)
>
> 처음엔 컴파일러 에러 메시지를 그대로 따라가세요. Rust 컴파일러는 **에러 메시지가 정말 친절**해서, 대부분 "이렇게 고쳐보세요"까지 알려줘요.

**🙋 학생**: 전체적으로 보면 Rust가 다른 언어보다 뭐가 좋은 거야?

> **선생님**: 한 문장으로: **"런타임에 터질 문제를 빌드 타임에 미리 잡아준다."**
>
> - C++: 자유롭지만 실수하면 런타임에 터짐 (segfault, double free, data race)
> - Java: GC가 대신 관리 → 안전하지만 성능 손해 (Stop-the-World)
> - Rust: 컴파일러가 소유권/빌림 검사 → 안전 + 성능 둘 다
>
> 빌드가 빡빡한 대신, **빌드만 통과하면 메모리 관련 런타임 버그가 없다**는 보장. 이게 Rust의 핵심 가치예요.

### 💡 핵심
- 90%는 `&T`(읽기 빌림)로 충분
- 소유권 에러 대처: & → &mut → clone() → Rc/Arc 순서로 시도
- 컴파일러 에러 메시지를 따라가면 대부분 해결됨
- Rust의 가치: 빌드 통과 = 메모리 안전 보장
